1) What is Arduino?

Arduino is an open‚Äësource ecosystem of microcontroller hardware (boards), software (IDE, libraries), and an enormous community. You write C/C++ code that runs directly on the microcontroller to interact with the real world‚ÄîLEDs, buttons, sensors, motors, and networks.

  • Beginners ‚Äî Quick wins with examples and libraries.
  • Hobbyists ‚Äî Home automation, robotics, wearables.
  • Engineers ‚Äî Fast, low‚Äëcost prototyping and education.

1.1) How Arduino Works

An Arduino board contains a microcontroller (e.g., ATmega328P) that executes your program from flash memory. You connect devices to its pins and read or control them in code.

  • Digital I/O: HIGH (5V/3.3V) or LOW (0V). pinMode(), digitalWrite(), digitalRead().
  • Analog Input: ADC converts voltages to numbers. Typical 10‚Äëbit on Uno ‚Üí 0‚Äì1023 for 0‚Äì5V.
  • PWM Output: Fake ‚Äúanalog‚Äù via fast on/off. analogWrite() (on PWM‚Äëcapable pins).
  • Clock & Timers: Timers power PWM, millis(), tone(), and advanced timing.
  • Bootloader & USB: Lets the IDE upload compiled code via USB without a separate programmer.
FYI:
  • Uno pins are 5V‚Äëtolerant; many modern boards are 3.3V only‚Äîalways check before wiring.
  • Each I/O pin has current limits (typically 20‚Äì40 mA max). Use a transistor or driver for motors/strips.
  • ‚ÄúAnalog‚Äù output via analogWrite() is PWM; true DAC exists on some boards (e.g., Due).

2) Choosing a Board

Pick based on voltage, GPIO count, memory, and connectivity:

Uno R3 / R4

Great starter. R3 (ATmega328P, 5V). R4 (Renesas RA4M1, 5V logic, more RAM/flash, faster).

Pros: community, shields. Cons: limited RAM (R3).

Nano (Classic / Every)

Tiny footprint for breadboards. USB‚ÄëC on newer variants; ideal for compact builds.

Mega 2560

Tons of pins and serial ports for complex projects (3D printers, CNC).

Due / MKR Series

3.3V logic. Due adds DACs and lots of horsepower. MKR adds connectivity options.

Portenta / Nicla

Pro‚Äëgrade performance, AI at the edge, industrial features.

3) Powering Arduino Safely

Reliable power prevents resets, USB brownouts, and hot regulators.

  • USB 5V: Best for development; limited current for peripherals.
  • Barrel Jack / Vin: Feed 7‚Äì12V into onboard regulator (linear on many boards ‚Üí heat!).
  • 5V / 3.3V Pins: Output from board‚Äôs regulator. Don‚Äôt back‚Äëfeed unless you know what you‚Äôre doing.
  • Separate supplies: For motors/LED strips, use a dedicated PSU and common ground.
Resistor choice for LEDs

Ohm‚Äôs Law: R = (Vsource ‚àí Vf) / I. Example: (5V ‚àí 2V) / 0.01A ‚âà 300Œ© ‚Üí pick 330Œ© or 220‚Äì1kŒ©.

Driving motors/relays

Use a transistor/MOSFET and a flyback diode on coils. Never power motors straight from a pin.

4) Programming Basics (Arduino C++)

Your sketch has two key functions: setup() runs once; loop() repeats forever. Use the IDE‚Äôs Library Manager for drivers and examples.

Blink with Commentary

// The built‚Äëin LED is on pin LED_BUILTIN (13 on Uno)
void setup() {
  pinMode(LED_BUILTIN, OUTPUT); // configure GPIO as output
}

void loop() {
  digitalWrite(LED_BUILTIN, HIGH); // LED on (5V)
  delay(500);                      // wait 500 ms
  digitalWrite(LED_BUILTIN, LOW);  // LED off (0V)
  delay(500);                      // repeat
}

Read a Potentiometer ‚Üí Dim an LED

const uint8_t POT = A0;        // analog input
const uint8_t LED = 9;         // PWM capable pin on Uno

void setup() {
  pinMode(LED, OUTPUT);
}

void loop() {
  int raw = analogRead(POT);        // 0..1023
  int pwm = map(raw, 0, 1023, 0, 255);
  analogWrite(LED, pwm);            // duty cycle 0..255
}
Why PWM isn‚Äôt ‚Äúreal‚Äù analog

PWM rapidly toggles a pin. After a low‚Äëpass effect (LED persistence or RC), it appears dimmer/brighter. For true analog voltage, you need a DAC (Due, some MKR/Portenta) or an external DAC.

5) Breadboarding & First Circuit

  1. Place a 5 mm LED on the breadboard (long leg = anode).
  2. Connect a 220‚Äì330 Œ© resistor from D13 to LED anode.
  3. Connect LED cathode (short leg) to GND.
  4. Upload the Blink sketch. If reversed, the LED won‚Äôt light‚Äîflip it.

Tip: Use color‚Äëcoded jumpers (red = V+, black = GND). Keep wires short and flat to reduce noise and confusion.

Debouncing a Button (no delay)

const uint8_t BTN = 2;         // use pin 2 with pullup
const uint8_t LED = 13;

bool led = false;
unsigned long last = 0;

void setup(){
  pinMode(BTN, INPUT_PULLUP);  // button to GND
  pinMode(LED, OUTPUT);
}

void loop(){
  static bool prev = HIGH;              // pulled up
  bool cur = digitalRead(BTN);
  unsigned long now = millis();
  if (cur != prev && (now - last) > 25) { // 25 ms debounce
    last = now; prev = cur;
    if (cur == LOW) {                    // pressed
      led = !led; digitalWrite(LED, led);
    }
  }
}

I¬≤C (Two‚Äëwire)

  • Pins: SDA/SCL (A4/A5 on Uno). Needs 4.7k‚Äì10k pull‚Äëups to 5V/3.3V.

  • Pins: MOSI/MISO/SCK + CS per device. Very fast streams (displays, SD cards).

  SPI.beginTransaction(SPISettings(8000000, MSBFIRST, SPI_MODE0));
  digitalWrite(CS, LOW);
  SPI.transfer(0x9F); // Read JEDEC ID on many chips
  digitalWrite(CS, HIGH);
  SPI.endTransaction();

UART (Serial) basics

TX‚ÜíRX, RX‚ÜíTX, and GND in common. Typical 9600‚Äì115200 baud. On Uno, pins 0/1 share the USB bridge‚Äîavoid collisions during upload.

7) Serial Monitor & Debugging Prints

Printing values to the Serial Monitor is the fastest way to debug logic and sensors.

#define DBG(...)  do{ Serial.print("[DBG] "); Serial.println(__VA_ARGS__); }while(0)

void setup(){
  Serial.begin(115200);
  while(!Serial){} // needed on some boards (Native USB)
  DBG("Boot OK");
}

void loop(){
  static unsigned long t = 0; 
  if (millis() - t > 1000) {
    t = millis();
    DBG(millis()); // print uptime every second
  }
}

Common Serial Pitfalls

  • Mismatched baud rate ‚Üí unreadable gibberish.
  • Printing too much too fast ‚Üí slows your loop.
  • On native USB boards, wait for the port to open.

Scheduling without delay()

struct Task{ unsigned long iv; unsigned long last; void (*fn)(); };

void ledBlink(){ digitalWrite(13, !digitalRead(13)); }
void sample(){ /* read sensors */ }

Task tasks[] = {
  {500, 0, ledBlink},
  {50,  0, sample},
};

void setup(){ pinMode(13, OUTPUT); }

void loop(){
  unsigned long now = millis();
  for (auto &t : tasks){ if (now - t.last >= t.iv){ t.last = now; t.fn(); } }
}

11) Best Practices

  • Use constants & types: const uint8_t for pins; avoid magic numbers.
  • Non‚Äëblocking timing: Prefer millis() over delay() for responsiveness.
  • State machines: Model behavior explicitly; easier to scale and debug.
  • Isolate hardware: Wrap sensors/actuators behind functions or classes.
  • Memory care: SRAM is scarce; store strings in flash via F() macro.
  • Safety first: Drivers for motors, level shifting for 3.3V devices, fuses for batteries.
Minimal Finite‚ÄëState Machine (FSM) Example 
enum State { IDLE, HEATING, COOLING };
State s = IDLE;

void loop(){
  int t = readTemp();
  switch (s){
    case IDLE:   if (t < 20) s = HEATING; else if (t > 28) s = COOLING; break;
    case HEATING: if (t >= 22) s = IDLE; break;
    case COOLING: if (t <= 26) s = IDLE; break;
  }
}
